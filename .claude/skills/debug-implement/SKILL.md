# デバッグ実行スキル

デバッグプランファイルを読み込み、Claude in Chrome でブラウザ操作による E2E 動作確認を行います。
各確認項目ごとにスクリーンショットと動作確認 GIF を撮影し、デバッグプランのステータスを更新します。

## 使い方

```bash
/debug-implement <debug-plan-ファイル名>
```

### 入力形式

| 形式         | 例                                                | 説明                                  |
| ------------ | ------------------------------------------------- | ------------------------------------- |
| ファイル名   | `debug-plan-20260311-1430.md`                     | outputs/debug-plans/ 配下のファイル名 |
| ファイルパス | `outputs/debug-plans/debug-plan-20260311-1430.md` | 相対パスまたは絶対パス                |

## 前提条件

- `debug-plan` スキルで作成されたデバッグプランファイルが存在すること
- デバッグ対象のアプリケーションが `localhost:3000` で起動していること
- Chrome ブラウザが起動しており、Claude in Chrome 拡張が有効であること

## 実行フロー

### 1. デバッグプランの読み込み

引数からデバッグプランファイルを特定し、読み込みます。

**ファイル検索の優先順位:**

1. 引数がフルパスの場合 → そのまま読み込み
2. 引数がファイル名の場合 → `outputs/debug-plans/` 配下を検索

読み込み後、以下の情報を抽出:

- **対象PR番号・URL**: 報告に使用
- **元ブランチ**: 正しいブランチにいるか確認
- **デバッグ用ブランチ**: 現在のブランチと一致するか確認
- **デバッグ項目一覧**: 新機能確認 + デグレーション確認
- **ユーザー介入が必要な項目**: スキップ対象として認識

### 2. 事前確認

```bash
# 現在のブランチがデバッグ用ブランチであるか確認
git branch --show-current
```

- デバッグ用ブランチにいない場合、ユーザーに確認の上チェックアウトする
- アプリケーションが起動しているか確認（localhost:3000 へのアクセス）

### 3. 出力ディレクトリの準備

```bash
# デバッグ実行結果の出力先を作成
mkdir -p outputs/debug-implements/<debug-plan-ファイル名（拡張子なし）>/screenshots
mkdir -p outputs/debug-implements/<debug-plan-ファイル名（拡張子なし）>/gifs
```

**ディレクトリ構造例:**

```
outputs/debug-implements/
└── debug-plan-20260311-1430/
    ├── screenshots/
    │   ├── 01-feature-login-before.png
    │   ├── 01-feature-login-after.png
    │   ├── 02-feature-dashboard.png
    │   └── ...
    ├── gifs/
    │   ├── 01-feature-login-flow.gif
    │   ├── 02-feature-dashboard-navigation.gif
    │   └── ...
    └── debug-report.md
```

### 4. ブラウザセッションの初期化

Claude in Chrome のツールを使用してブラウザ操作を開始します。

```
1. tabs_context_mcp でブラウザの現在の状態を確認
2. 必要に応じて tabs_create_mcp で新しいタブを作成
3. navigate で localhost:3000 にアクセス
4. read_page で初期状態を確認
```

### 5. デバッグ項目の実行

デバッグプランの各項目を順番に実行します。

#### 5-A. 各項目の実行手順

1. **項目の確認手順を読み取る**
2. **GIF記録を開始**: `gif_creator` でブラウザ操作を記録
   - GIF ファイル名: `<項番>-<カテゴリ>-<項目の短縮名>.gif`
   - カテゴリ: `feature`（新機能確認）または `regression`（デグレ確認）
3. **ブラウザ操作を実行**: 確認手順に従い、以下のツールを使い分ける
   - `navigate`: ページ遷移
   - `form_input`: フォーム入力
   - `computer`: クリック等の操作
   - `find`: 要素の検索
   - `read_page`: ページ内容の確認
   - `get_page_text`: テキスト取得
   - `javascript_tool`: JavaScript 実行（DOM操作・状態確認）
4. **スクリーンショットを撮影**: 操作の前後や重要な状態でスクリーンショットを取得
   - `computer` ツールの screenshot 機能を使用
   - ファイル名: `<項番>-<カテゴリ>-<状態説明>.png`
5. **GIF記録を保存**: 操作完了後に GIF を保存
6. **結果を判定**: 期待結果と実際の結果を比較

#### 5-B. 結果判定の基準

| ステータス    | 条件                             |
| ------------- | -------------------------------- |
| ✅ 確認OK     | 期待結果と実際の表示/動作が一致  |
| ❌ 不具合あり | 期待結果と異なる動作、エラー発生 |
| ⚠️ 要確認     | 自動判定が困難、目視確認が必要   |
| 🔄 スキップ   | ユーザー介入が必要な項目         |

#### 5-C. ユーザー介入が必要な項目の扱い

デバッグプランで「ユーザー介入が必要」と指定された項目は:

- ステータスを `🔄 スキップ` に設定
- 備考欄に理由を記載（「メール認証が必要」等）
- スキップした旨をレポートに記録

### 6. デバッグレポートの作成

全項目の実行完了後、`debug-report.md` を生成します。

**出力先:**

```bash
outputs/debug-implements/<debug-plan-ファイル名（拡張子なし）>/debug-report.md
```

**レポートテンプレート:** `.claude/templates/debug-report-template.md`

### 7. デバッグプランファイルの更新

デバッグ実行結果をもとに、元のデバッグプランファイルのステータス列を更新します。

### 8. ユーザーへの完了報告と次ステップの提案

```
デバッグ実行が完了しました:

- デバッグプラン: <ファイル名>
- 対象PR: #<PR番号> <PRタイトル>
- 実行結果: ✅ N件 / ❌ N件 / ⚠️ N件 / 🔄 N件
- レポート: outputs/debug-implements/<plan名>/debug-report.md
- スクリーンショット: outputs/debug-implements/<plan名>/screenshots/
- GIF: outputs/debug-implements/<plan名>/gifs/

> `/clear` をして、`/debug-pr-create <debug-plan-ファイル名>` を実行してください
```

## 注意事項

- **コミット・プッシュは行わない**: ユーザーの許可なくコミット・プッシュは行いません
- **localhost:3000 のみ操作**: 外部サイトへのアクセスは行わない
- **破壊的操作の回避**: データベースのデータを壊す可能性のある操作は慎重に行い、テストデータを使用する
- **GIF は操作単位で分割**: 1つのデバッグ項目が長い場合でも、意味のある操作単位でGIFを分割して見やすくする
- **エラー発生時の継続**: 1つの項目でエラーが発生しても、残りの項目は継続して実行する
- **console エラーの確認**: 各操作後に `read_console_messages` でコンソールエラーがないかも確認し、エラーがあれば備考欄に記載する
- **ページ遷移ごとにスクリーンショット**: ページ遷移が発生するたびにスクリーンショットを保存し、エビデンスを残す
