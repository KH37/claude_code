# デバッグプラン作成スキル

PRの差分を分析し、E2Eの動作確認項目（デグレーション含む）を作成します。

## 使い方

```bash
/debug-plan <PR番号>
/debug-plan <PR URL>
```

### 入力形式

| 形式   | 例                                       | 説明                     |
| ------ | ---------------------------------------- | ------------------------ |
| PR番号 | `123`                                    | 現在のリポジトリのPR番号 |
| PR URL | `https://github.com/owner/repo/pull/123` | 完全なPR URL             |

## 実行フロー

### 1. PR情報の取得

```bash
# PR詳細を取得
gh pr view $ARGUMENTS --json number,title,url,headRefName,baseRefName,body,files,additions,deletions

# PRの差分を取得
gh pr diff $ARGUMENTS
```

取得した情報から以下を抽出:

- **PR番号**: デバッグプラン・ブランチ名に使用
- **PRタイトル**: デバッグプランの基本情報に記載
- **PR URL**: デバッグプランの基本情報に記載
- **headRefName**: 元ブランチ名（デバッグ用ブランチの派生元）
- **変更ファイル一覧**: デバッグ項目の分析に使用
- **差分内容**: デバッグ項目の詳細分析に使用

### 2. 現在のブランチを確認し、PRのブランチに切り替え

```bash
# 現在のブランチを確認
git branch --show-current

# PRのブランチをチェックアウト（フォークPRにも対応）
gh pr checkout <PR番号>
```

### 3. デバッグ用ブランチの作成

**ブランチ命名規則:**

```bash
debug/<PR番号>-<元ブランチの短縮名>
```

**例:**

- 元ブランチ: `feat/123-add-login` → デバッグ用ブランチ: `debug/456-feat-123-add-login`
  - `456` はPR番号

```bash
git checkout -b debug/<PR番号>-<元ブランチの短縮名>
```

### 4. PRの差分を分析してデバッグ項目を作成

PRの差分を以下の観点で分析し、デバッグ項目を作成します:

#### 4-A. 新機能・変更機能の動作確認項目

変更内容から直接的な動作確認項目を洗い出します:

- **UI変更**: 画面表示、レイアウト、レスポンシブ対応
- **API変更**: エンドポイントの動作、リクエスト/レスポンス
- **ロジック変更**: 条件分岐、計算結果、状態遷移
- **データベース変更**: データの保存・取得・更新・削除
- **認証/認可**: アクセス制御、権限チェック

#### 4-B. デグレーション確認項目

変更の影響範囲から、既存機能への悪影響がないか確認する項目を洗い出します:

- 変更ファイルを使用している**他の機能**
- 変更されたAPIを呼び出している**画面やコンポーネント**
- 共通コンポーネントの変更による**他画面への影響**
- 設定値の変更による**既存動作への影響**

#### 4-C. ユーザー介入が必要な項目の特定

以下のような項目はAIエージェントでは自動確認が困難なため、明記します:

- メール認証・SMS認証
- 外部サービス連携（OAuth、決済など）
- 物理デバイス依存（カメラ、GPS、NFC等）
- CAPTCHA
- ファイルアップロード（実ファイルが必要な場合）
- ブラウザ通知・プッシュ通知

### 5. デバッグプランファイルの作成

テンプレートファイルを読み込み、デバッグプランを生成します。

```bash
# テンプレート読み込み
cat .claude/templates/debug-plan-template.md
```

**出力先:**

```bash
outputs/debug-plans/debug-plan-<YYYYMMDD>-<HHmm>.md
```

**ファイル名の例:**

- `debug-plan-20260311-1430.md`

テンプレートの各プレースホルダーを実際の値で置き換え、分析結果をもとにデバッグ項目テーブルを埋めます。

### 6. ユーザーへの完了報告と次ステップの提案

デバッグプラン作成完了後、以下を報告します:

```bash
デバッグプランを作成しました:

- ファイル: outputs/debug-plans/debug-plan-<YYYYMMDD>-<HHmm>.md
- 対象PR: #<PR番号> <PRタイトル>
- デバッグ用ブランチ: debug/<PR番号>-<元ブランチの短縮名>
- デバッグ項目数: 新機能確認 N件 / デグレ確認 N件
- ユーザー介入必要: N件

> `/clear` をして、`/debug-implement debug-plan-<YYYYMMDD>-<HHmm>.md` を実行してください
```

## 注意事項

- **コミット・プッシュは行わない**: このスキルはデバッグプランの作成とブランチ作成のみを行います
- **デバッグ項目は具体的に**: 「画面を確認する」ではなく「ログイン画面でメールアドレスとパスワードを入力し、ログインボタンを押下した際にダッシュボードに遷移すること」のように具体的な手順と期待結果を記載する
- **デグレーション確認は変更の影響範囲を考慮**: 直接変更したファイルだけでなく、そのファイルを参照している箇所も確認対象とする
- **claude in chromeで実行可能な粒度**: デバッグ項目はブラウザ操作で確認できる粒度で記載する（後続の `debug-implement` でclaude in chromeが実行するため）
