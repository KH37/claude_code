ユーザから提供された「$ARGUMENTS」（議事録、会議メモ、要件文書等）を分析し、実行可能なタスクを抽出してGitHub issueを作成します。

## 手順詳細

### 1. 文書の読み込みと理解
- 「$ARGUMENTS」がファイルパスの場合はファイルを読み込む
- テキストが直接渡された場合はその内容を分析
- 文書の種類を判定（議事録、要件定義書、ブレストメモ等）

### 2. タスクの抽出
以下の観点から実行可能なタスクを識別:

#### プログラミング実装タスク
- 新機能の開発
- バグ修正
- リファクタリング
- テストコード追加
- パフォーマンス改善

#### ドキュメント・設計タスク
- 仕様書作成
- API設計書作成
- ドキュメント更新
- README更新
- アーキテクチャ図作成

#### 調査・検証タスク
- 技術調査（ライブラリ選定等）
- 実現可能性検証（PoC）
- パフォーマンス調査
- セキュリティ監査
- 競合調査

#### 運用・インフラタスク
- デプロイ設定
- CI/CD設定
- 監視設定
- ログ設定
- インフラ構築

#### その他のタスク
- レビュー依頼
- ミーティング設定
- 外部連携調整
- ユーザーヒアリング
- データ移行

### 3. タスクの構造化
抽出した各タスクについて以下を整理:
- **タスク名**: 簡潔で明確な名称
- **カテゴリ**: 実装/ドキュメント/調査/運用/その他
- **優先度**: 議事録内の文脈から推測（高/中/低）
- **期限**: 明記されていれば記録
- **担当候補**: 議事録に記載があれば記録
- **依存関係**: 他タスクとの前後関係
- **背景**: なぜこのタスクが必要か
- **成果物**: 何が完成すればタスク完了か

### 4. タスク一覧の提示
抽出したタスクをユーザに提示し、確認を求める:
```
## 抽出されたタスク一覧

### プログラミング実装（3件）
1. [高] ユーザー認証API実装
   - 背景: セキュリティ強化のため
   - 期限: 来週金曜日
   
2. [中] ダッシュボード画面のレスポンシブ対応
   - 背景: モバイルユーザー増加
   
3. [低] エラーログの詳細化
   - 背景: デバッグ効率化

### ドキュメント（2件）
4. [高] API仕様書v2作成
   - 背景: 外部連携のため必須
   - 依存: タスク1完了後
   
5. [中] READMEのセットアップ手順更新
   - 背景: 新メンバーのオンボーディング改善

### 調査・検証（1件）
6. [中] 画像処理ライブラリの比較検証
   - 背景: パフォーマンス改善のため

---
これらのタスクでissueを作成しますか？
- すべて作成する → 「はい」と入力
- 一部のみ選択 → タスク番号を指定（例: 1,2,4）
- 修正が必要 → 修正内容を指示
```

### 5. 不足情報の確認
タスクごとに以下が不明確な場合はユーザに質問:
- 具体的な実装方針
- 技術スタックの選定
- 成果物の詳細
- 優先度や期限
- 担当者

**複数タスクの場合**: まとめて質問してから一括でissue作成

### 6. GitHub issueの作成
ユーザの確認後、選択されたタスクごとに`gh issue create`コマンドで個別issueを作成。

**タイトル**: `[カテゴリ] タスク名`の形式

**本文**:
```
## 概要
[タスクの内容を簡潔に]

## 背景・目的
[議事録から抽出した背景・理由]

## 実装内容 / 作業内容
- [ ] [具体的な作業1]
- [ ] [具体的な作業2]
- [ ] [具体的な作業3]

## 成果物
[何が完成すればこのタスクは完了か]

## 関連情報
- 元議事録: [ファイル名または日付]
- 議事録での言及: 
  > [該当部分の引用]

## 依存関係
[他のissueとの依存関係]
- 前提: #[issue番号]
- 関連: #[issue番号]

## 期限
[期限が明記されている場合]

## 参考情報
[関連するドキュメント、リンク等]
```

**ラベル設定**: カテゴリに応じて適切なラベルを付与
- 実装 → `enhancement`, `feature`
- バグ → `bug`
- ドキュメント → `documentation`
- 調査 → `research`, `investigation`
- 運用 → `infrastructure`, `deployment`

**マイルストーン**: 期限やプロジェクトフェーズに応じて設定

### 7. 作成完了の報告
```
## 作成完了したissue

✅ #123 [実装] ユーザー認証API実装
✅ #124 [実装] ダッシュボード画面のレスポンシブ対応
✅ #125 [ドキュメント] API仕様書v2作成
✅ #126 [調査] 画像処理ライブラリの比較検証

## 推奨される実装順序
1. #123（他タスクの前提）
2. #126（並行可能）
3. #125（#123完了後）
4. #124（並行可能）

## 次のアクション
- 各issueに担当者をアサインしてください
- 優先度に応じてマイルストーンを設定してください
```

## その他
- issueの内容は日本語で書く
- 議事録の文脈を最大限活用し、背景や理由を明確に記載
- タスクとして不明確なものは「要確認」として別途整理
- 「検討する」「考える」といった曖昧な表現は、具体的な調査タスクに変換
- 想定外の作業やエラーが発生した場合はユーザに相談
- 議事録から10個以上のタスクが抽出された場合は、優先度の高いものを先に提示
- プログラミング以外のタスク（会議設定、メール送付等）で、GitHub issueとして管理すべきでないものは除外し、別途リスト化してユーザに提示
