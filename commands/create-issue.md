ユーザから要求された「$ARGUMENTS」に対して以下の手順で課題を作成します。

## 手順詳細

### 1. タスクの分割
- 「$ARGUMENTS」を分析し、複数のタスクが含まれているか確認
- 含まれている場合は、各タスクを個別に識別して分離
- 各タスク間の依存関係があれば記録

### 2. プロジェクト構造の理解
- リポジトリのREADME、package.json、requirements.txt等を確認してプロジェクトの概要を把握
- 既存のissueやPRを確認して、プロジェクトの命名規則やissueフォーマットを理解
- .github/ISSUE_TEMPLATE/がある場合はそのテンプレートに従う

### 3. 各タスクごとに以下を実施

#### 3-1. 要件の分析と整理
- タスクの内容を分析し、以下を明確化:
  - 何を実現したいのか（目的）
  - なぜ必要なのか（背景・理由）
  - どのように実装するか（実装方針の概要）
  
#### 3-2. 実装箇所の特定
- 関連するファイルやディレクトリを検索
- 既存の類似実装を探して参考にする
- 影響範囲を特定（フロントエンド、バックエンド、データベース等）

#### 3-3. 不足情報の確認
以下のような情報が不明確な場合はユーザに質問:
- 技術スタック・ライブラリの選定
- UI/UXの詳細仕様
- パフォーマンス要件
- セキュリティ要件
- テスト方針
- 優先度・期限
- タスク間の依存関係や実装順序

**複数タスクの場合**: すべてのタスクについて不足情報を一度にまとめて質問してから、issue作成に進む

#### 3-4. GitHub issueの作成
`gh issue create`コマンドを使用して**タスクごとに個別のissue**を作成。以下の構造で記載:

**タイトル**: 簡潔で具体的に（プロジェクトの命名規則に従う）

**本文**:
```
## 概要
[実現したいことを簡潔に]

## 背景・目的
[なぜこの機能が必要か]

## 実装内容
- [ ] [具体的なタスク1]
- [ ] [具体的なタスク2]
- [ ] [具体的なタスク3]

## 実装箇所（候補）
- `path/to/file1.ext` - [役割]
- `path/to/file2.ext` - [役割]

## 技術的検討事項
[実装時に考慮すべきポイント]

## 依存関係
[他のissueとの依存関係がある場合は記載]
- 前提: #[issue番号]
- 関連: #[issue番号]

## 参考情報
[関連するissue、ドキュメント、外部リンク等]
```

**ラベル・アサイン**: プロジェクトの慣習に従って設定

### 4. 作成完了の報告
- 作成したissueの一覧をユーザに提示
- 各issueの番号とタイトルを明記
- タスク間に依存関係がある場合は、推奨される実装順序を提案

## その他
- issueの内容は日本語で書く
- 想定外の作業やエラーが発生した場合はユーザに相談
- プロジェクト構造を把握できない場合や、実装箇所が特定できない場合は、その旨をissueに記載してユーザに確認を求める
- 複数の実装アプローチが考えられる場合は、選択肢を提示してユーザに判断を仰ぐ
- タスクが5個以上ある場合は、まずタスクの一覧を提示してユーザに確認を取ってから作成を進める
